## Architecture Decisions

### wcagValue as a key in `Record<string, string>` (not a new interface)

The `primitiveReferences` type remains `Record<string, string>`. Adding `wcagValue` as a key is already type-compatible. A dedicated interface was considered but rejected because:
- It would require updating every token definition's type annotation
- The `Record<string, string>` pattern is used throughout the codebase for flexible key sets
- JSDoc documentation is sufficient to communicate the convention

### WCAG overrides at orchestration level, not format generator level

The WCAG override block is generated by `TokenFileGenerator` (orchestrator), not by `WebFormatGenerator`/`iOSFormatGenerator`/`AndroidFormatGenerator`. This is because:
- The override is a separate CSS selector / code section, not a modification to individual token formatting
- The format generators' `formatSingleReferenceToken()` correctly uses `primitiveReferences.value` for standard output
- Keeping the override logic centralized avoids duplicating theme-awareness across 3 format generators

### DTCG/Figma guard rails as throw (not skip)

The guard rails throw errors rather than silently skipping `wcagValue` tokens because:
- Silent skipping would produce incomplete output without warning
- The error message identifies the specific token and recommends a follow-up spec
- This aligns with the design doc's DTCG/Figma gap analysis (Requirement 11)
